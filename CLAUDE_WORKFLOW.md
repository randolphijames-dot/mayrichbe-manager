## ❌ 曾经犯过的严重错误

### 错误案例 #1：未能主动发现系统已有功能

**时间**：2026-03-12
**场景**：用户说"我用模拟手机，不用 adspower"

**我的错误做法**：
1. 直接推荐用户安装 AdsPower
2. 讨论 Android 模拟器方案
3. 等用户提醒"有比特浏览器选项"才去检查代码
4. 浪费了 30+ 分钟

**问题根源**：
- ❌ 读了代码但没有建立完整功能清单
- ❌ 思维定式：看到 Instagram 就想到 AdsPower
- ❌ 没有先搜索系统支持什么，就开始推荐
- ❌ 反应式思维，而非主动式思维

**正确做法应该是**：
```bash
# 用户说不用 AdsPower 时，立即执行：
grep -r "BrowserType\|browser_type" backend/
# 1 秒钟发现：ADSPOWER、BITBROWSER、NONE

# 然后回复：
"系统支持 3 种浏览器方式：
1. AdsPower（指纹浏览器）
2. 比特浏览器（指纹浏览器）
3. 不使用浏览器（instagrapi 模拟手机 App）

你现在用的是哪种？"
```

**用户反馈**：
> "我的天，你这都不知道吗？你不是已经读了整个底层代码吗？"

**教训**：
- 读代码 ≠ 理解系统
- **任何推荐之前，先搜索已有功能**
- **列出所有选项，让用户选择**

## ✅ 正确的工作流程

### 流程 1：任务开始时的系统理解

```bash
# Step 1: 建立完整功能清单（第一次对话时必做）
grep -r "class.*Enum" backend/app/models/
grep -r "router = APIRouter" backend/app/api/
ls -la backend/app/services/

# Step 2: 总结关键功能
echo "系统支持的功能：
- 平台：Instagram, YouTube
- 浏览器：AdsPower, BitBrowser, None
- 任务：定时发布、立即发布、批量发布
- ..."
```

**输出示例**：
```markdown
## 系统功能清单

### 支持的平台
- Instagram
- YouTube

### 支持的浏览器类型
- AdsPower（指纹浏览器）
- 比特浏览器（指纹浏览器）
- None（instagrapi 模拟手机）

### 发布方式
- 立即发布（instant=true）
- 定时发布（scheduled_at）
- 批量发布（看板模式）
```

---

### 流程 2：用户提问时的响应模式

```
用户问题
  ↓
先搜索系统已有功能（grep/read）
  ↓
列出所有可用选项
  ↓
询问用户需求/偏好
  ↓
提供具体方案
```

**示例**：

```
❌ 错误：
用户："我不用 X"
我："那你要不要用 Y？我帮你配置 Y"

✅ 正确：
用户："我不用 X"
我：grep 搜索替代方案
我："系统支持 X、Y、Z 三种方式，你现在用的是哪个？"
```

### 流程 3：代码修改前的检查清单

在修改代码或提供方案之前，必须回答：

- [ ] 我是否已经搜索了相关的现有代码？
- [ ] 系统是否已经有类似功能？
- [ ] 我是否列出了所有可用选项？
- [ ] 我是否询问了用户的偏好？
- [ ] 我的建议是基于代码事实，还是基于假设？

## 🎯 核心原则

### 原则 1：先搜索，再回答
```
任何推荐之前：
grep -r "关键词" → 理解现状 → 列出选项 → 询问用户
```

### 原则 2：提供选项，而非单一推荐
```
❌ "你应该用 X"
✅ "系统支持 A、B、C，你更倾向哪个？"
```

### 原则 3：承认不确定，而非假设
```
❌ "这个功能不存在"（未经搜索）
✅ "让我搜索一下... 发现系统已经支持！"
```

### 原则 4：读代码要建立完整图谱
```
读取文件 → 总结功能 → 建立清单 → 存储在记忆中
```

### 原则 5：反应式 → 主动式
```
❌ 等用户提醒才发现功能
✅ 主动列出系统所有能力
```

## 📋 每次对话开始的检查清单

### Phase 1：系统理解（首次对话）
- [ ] 执行 `grep -r "Enum\|APIRouter" backend/`
- [ ] 列出所有支持的平台、浏览器、功能
- [ ] 建立完整功能清单

### Phase 2：用户需求理解
- [ ] 用户说"不用 X"时，立即搜索替代方案
- [ ] 列出所有选项，询问用户偏好
- [ ] 确认用户当前配置状态

### Phase 3：方案提供
- [ ] 基于代码事实，而非假设
- [ ] 提供具体步骤和示例
- [ ] 预判可能的问题

### Phase 4：执行验证
- [ ] 代码修改后测试
- [ ] 查看日志确认结果
- [ ] 向用户确认是否符合预期

## 💡 给自己的提醒

**永远记住**：
- 我读了代码，但可能没理解系统
- 用户比我更了解他们的需求
- **搜索优先，假设最后**
- **选项优先，推荐最后**
- **事实优先，经验最后**

**思考顺序**：
```
1. 系统支持什么？（grep/read）
2. 用户需要什么？（询问）
3. 如何结合？（方案）
```

**而不是**：
```
1. 我觉得应该用 X（假设）
2. 推荐用户用 X（推销）
3. 发现系统已经支持 Y（浪费时间）
```
